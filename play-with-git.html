Git Conflicts & Fast Forward

  • If two developers have their repo pointing to the same commit-id and both change stuff locally conflicts can happen
  • Example same file, different line
  • Developer a changes a.txt line 1 and commits locally + pushes
  • Developer b changes a.txt line 4 and commits locally + pushes (wihtout pull) > Result: conflict
  • Developer b will have to do a pull first, which results in another fetch+merge that can be commited
  • Automatic merge will work
    • Merge strategy used: recursive = integrates both changes
  • Example different files
  • Example same file, same line
  • Developer a changes a.txt line 5 and commits locally + pushes
  • Developer b changes a.txt line 5 and commits locally + pushes
  • Developer b will have to do a pull first, which results in another fetch+merge that can be commited
  • Automatic merge will FAIL
    • Manual merge has to be done
    • HEAD is the local version (local HEAD)
  • Mergetools can help
  • git mergetool --tool-help
  • git config --list
  • git mergetool --tool=meld (brew install meld)

Git Branches

  • Allow to work on features independently
  • Allow to work on multiple features concurrently
  • Allow stability (play around without destroying main codebase)
command comment
git checkout -b feature1 Create a new branch
git add . && git commit -m 'msg' commit on feature branch
git checkout main Checkout local main branch
git pull Bring main up2date
git merge feature1 Merge feature1 into master
git merge --squash feature2 Merge all commits from feature2 into a single commit

| | | | git checkout -b develop | Create new develop branch | | git switch -c develop | Newer version of checkout, -c==create | | git push --set-upstream origin develop | Push branch upstream | | git checkout -b dev-feature4A develop | Developer A | | git checkout -b dev-feature4B develop | Developer B |

Git Merge vs Rebase

  • Merge shows different branches and creates a new Merge Commit
  • Forward Based behaviour, history of both (e.g. main & feature1) are still visible = history is cluttered
  • Rebase shows a linear commit history
  • Rebase has issues with pull requests
  • Rebase = rewrite history
  • Rebase = ReBase, update history onto another branch
  • Example
    • Common base of main and feature is Commit#M2
    • Feature adds Commit#F1,ommit#F2,ommit#F3
    • Main adds Commit#M3,Cmmit#M4
  • Popular rebase example:
  • Developer A&B bases his wor on develop branch
  • Developer A works on feature1
  • Developer B works on feature2
  • Developer A finishes early and integrates feature1 into develop branch + pushes upstream
  • Developer B finishes his work on feature2
  • Developer B pulls changes from developer branch
  • Before merging his changes into develop branch, rebases his feature2 onto "current" develop branch state
  • Rebase on local project OK, rebase on public project NoGo
