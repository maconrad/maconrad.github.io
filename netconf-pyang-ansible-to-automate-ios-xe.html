How to setup the Switch for Netconf

Basic functionality of Netconf is simply achieved via using netconf-yang command. However the great benefit of netconf comes with using validation and rollback, which is a feature in DB world as ATOMIC operations. Either all-or-nothing. This can be achieved by netconf using the datastore. Furthermore an ACL should be applied to restrict connectivity to the management Hosts.

hostname SWITCH
no service config
username admin privilege 15 algorithm-type scrypt secret Admin123
enable secret Admin123
service password-encryption
ip domain name info.lab
system mtu 9100

line vty 0 15
 login local
 transport input ssh
 transport preferred none

crypto key generate rsa modulus 2048
ip ssh version 2
ip scp server enable

vrf definition Mgmt
 rd 1:1
 address-family ipv4

int gig0
 vrf forwarding Mgmt
 ip address 10.10.10.200 255.255.255.0

ip route vrf Mgmt 0.0.0.0 0.0.0.0 10.10.10.1

ip access-list extended ACL_NETCONF_PERMIT
 permit tcp 10.20.20.0 0.0.0.255 host 10.10.10.200 eq 830

netconf-yang
netconf-yang feature candidate-datastore
netconf-yang ssh ipv4 access-list name ACL_NETCONF_PERMIT

event manager applet catch_all
 event cli pattern ".*" sync no skip no
 action 1 syslog msg "$_cli_msg"

Using Ansible Netconf Module

For the sake of completness and easyness of usage, installation of the module and usage is quickly summarized: - Setup python venv using on of the known tools like anaconda, virtualenvwrapper or std-lib bundles venv module - Installation of ansible - Setup of inventory - Installation of Netconf via Netcommons Ansible Galaxy Collection

Ansible installation

# Create virtualenv using virtualenvwrapper
mkvirtualenv netconf-ansible
workon netconf-ansible

python -m venv netconf-env
source netconf-env/bin/activate

# Install ansible (v.5.5.0 and ansible-core 2.12 in my case)
pip install ansible
# Paramiko required by ansible.netcommon.network_cli
pip install paramiko

# Verify python3 is used in the backend
ansible --version | grep "python version"

# Quick test of ansible installation
ansible localhost -m ping

# Validate what Cisco enhancements are already installed
ansible-galaxy collection list | grep cisco

Note OSX

On OSX when using pyenv, the interpreter points directly to pyenv installed version. Therefore the modules/packages search path is not correctly added. Therefore e.g. paramiko is not found when executing ansible. The easy way is to install the module globally, which would pollute the system. The better way is to adjust the python search path.

# Get current search path
ansible --version | grep "configured module search path" 

# Get path of venv
source netconf-env/bin/activate
# Start interactive shell and look for site-package path if unkown
python
 import sys
 print(sys.path)

# Add package path to ansible.cfg library=... using colon (:) notation for multiple
## See below

Note MAC: ansible 127.0.0.1 -m shell -a "python -c 'import sys; print(sys.path)'" --connection=local

Validate Switch Connectivity without Netconf

We setup ansible, an inventory containing our switch and configure a NTP server using a simple playbook. It is always good to first do some basic testing with a simple playbook to see everything works as expected.

# ansible.cfg
[defaults]
inventory = inventory
interpreter_python=/Users/conrad/<...>/netconf_play/netconf-env/bin/python
library = /Users/conrad/.ansible/plugins/modules:/usr/share/ansible/plugins/modules:/Users/conrad/<...>/netconf_play/netconf-env/lib/python3.9/site-packages

Configure a basic inventory:

# inventory
[routers]
10.10.10.200

[routers:vars]
ansible_connection=ansible.netcommon.network_cli
ansible_become=yes
ansible_become_method=enable
ansible_network_os=cisco.ios.ios
ansible_user=admin
ansible_password=supersecret

Setup the playbook:

---
- name: Configure the banner
  gather_facts: false
  hosts: routers
  tasks:

    - name: Configure the login banner
      cisco.ios.ios_banner:
        banner: login
        text: |
          ---- Fully automated Box ----
          - Do NOT change manually.   -
          -----------------------------
        state: present

# Update collection to latest version
ansible-galaxy collection install cisco.ios

# Syntax Check, lint and run Playbook
ansible-lint 1_play_banner.yaml
ansible-playbook 1_play_banner.yaml --syntax-check
ansible-playbook 1_play_banner.yaml 

# In case of errors
ansible-playbook 1_play_banner.yaml -vvv
export ANSIBLE_DEBUG=1

Basic Netconf Automation - NTP

Ansible netconf module has been moved (like many modules) to be part of a collection, namely the ansible.netcommon collection.

# Install/update netcommon, as Netconf part of netcommon module
ansible-galaxy collection list | grep netcommon

# Install netconf python library
pip install ncclient
pip install netconf-console
pip install netconf-console2
ppip install jxmlease

netconf-console --host 10.10.10.200 --port 830 --user admin --password xy --db running --get-config --xpath /native

Adjust the Connection parameter in inventory:

# inventory
...
[routers:vars]
#ansible_connection=ansible.netcommon.network_cli
ansible_connection=ansible.netcommon.netconf

And finally the playbook, here to set an NTP Server on a router that does not support candidate datastore.

- hosts: routers
  connection: ansible.netcommon.netconf
  gather_facts: no
  tasks:
    - ansible.netcommon.netconf_config:
        target: running
        content: |
          <config xmlns="urn:ietf:params:xml:ns:netconf:base:1.0" xmlns:nc="urn:ietf:params:xml:ns:netconf:base:1.0">
            <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
              <ntp>
                <server xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-ntp" operation='replace'>
                  <server-list>
                    <ip-address>1.1.1.1</ip-address>
                  </server-list>
                </server>
              </ntp>
            </native>
          </config>

Some of the supported operations of thet netcommons.netconf module are:

- format (xml or json)
- get_filter
- confirm, lock, save, confirm_commit, validate, commit, delete, backup 
- content (defaults to xml)
- source_datastore (source), target (datastore).

Automation - Getting Values

The easiest one would be getting all configuraton:

- hosts: routers
  connection: ansible.netcommon.netconf
  gather_facts: no
  tasks:
    - ansible.netcommon.netconf_get:
        source: running

Several aditioinal options exist like adding a filter (in XML or XPath style) or output in json.

- hosts: routers
  connection: ansible.netcommon.netconf
  gather_facts: no
  tasks:
    - ansible.netcommon.netconf_get:
        source: running
        filter: /native/logging/history
        display: json

Automation Plus - RPCs & Notifications

RPCs are the way netconf implements actions. In fact edit-config is just an rpc of the ietf-netconf.yang specification. Custom RPCs are also defined in the yang models themselfes.

# Get a list of all rpcs per module
grep rpc ./*

# List of IOS XE rpcs
pyang -f tree Cisco-IOS-XE-rpc.yang --tree-depth=3

Example use cases:

- Reload routers, start traffic captures,
- Clear arp caches and bgp processes
- Register to smart licensesing
- Start aguestshell, enable debug
- etc. etc.

- hosts: routers
  connection: ansible.netcommon.netconf
  gather_facts: no
  tasks:
    - ansible.netcommon.netconf_rpc:
        xmlns: http://cisco.com/ns/yang/Cisco-IOS-XE-rpc
        rpc: clear
        content: |
          <arp-cache>
           <vrf>Mgmt 10.240.72.144</vrf>
          </arp-cache>

Using Python

The nccclient module can be used directly. The call is very similar to the ones shown above, as in the end, it is netconf format. Important to note are the supported device types, which are:

- Cisco: csr/nexus/iosxr/iosxe
- Non-cisco: junos/huawei/alu(alcatel-lucent)/hpcomware(HP)
- default

Get-Config

A very simple example getting configuration would be:

from ncclient import manager


def main():
    # Device dictionary that provides key/value connection information
    device = {"ip": "10.10.10.200", "port": "830", "platform": "csr",}

    # ncclient manager instantiation for csr
    with manager.connect(host=device['ip'], port=device['port'], username='admin',
                         password='xy', hostkey_verify=False,
                         device_params={'name': device['platform']},
                         look_for_keys=False, allow_agent=False) as m:

        # Filter using top-level container namespace and node matching
        int_filter = '''
                    <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
                        <interface>
                            <name>GigabitEthernet1</name>
                        </interface>
                    </interfaces>
                    '''

        # get-config RPC against the running datastore using a subtree filter
        reply = m.get_config('running', filter=('subtree', int_filter))

        # Print RPC reply
        print(reply)


if __name__ == '__main__':
    main()

Edit-Config

A very simple example to configure an OSPF Process would be:

from ncclient import manager


def main():
    device = {"ip": "10.10.10.200", "port": "830", "platform": "csr",}

    with manager.connect(host=device['ip'], port=device['port'], username='admin',
                         password='xy', hostkey_verify=False,
                         device_params={'name': device['platform']},
                         look_for_keys=False, allow_agent=False) as m:

        rpc = '''
                <config xmlns="urn:ietf:params:xml:ns:netconf:base:1.0" xmlns:nc="urn:ietf:params:xml:ns:netconf:base:1.0">
                    <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
                        <router>
                            <ospf xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-ospf">
                                <id>100</id>
                                <router-id>1.1.1.1</router-id>
                                <network>
                                    <ip>10.1.1.0</ip>
                                    <mask>0.0.0.3</mask>
                                    <area>0</area>
                                </network>
                            </ospf>
                        </router>
                    </native>
                </config>
            '''

        reply = m.edit_config(rpc, target='running')
        print(reply)


if __name__ == '__main__':
    main()

Sometimes we want to look at the yang models directly, to understand what can be configured or to generate stub data (see next chapter). Instead of downloading the model from the git repository, it can be directly downloaded from the device.

import re
from ncclient import manager

# Context manager keeping ncclient connection alive for the duration of
# the context.
m = manager.connect(
    host='10.10.10.200',   # IP address of the device
    port=830,              # Port to connect to
    username='admin',      # SSH Username
    password='xy.',        # SSH Password
    hostkey_verify=False   # Allow unknown hostkeys not in local store
)
capabilities = []
# Write each capability to console
for capability in m.server_capabilities:
    # Print the capability
    print(f"Capability: {capability}")
    # Store the capability list for later.
    capabilities.append(capability)


# Sort the list alphabetically.
capabilities = sorted(capabilities)

# List of modules that we store for use later
modules = []

# Iterate server capabilities and extract supported modules.
for capability in capabilities:
    # Scan the capabilities and extract modules via this regex.
    # i.e., if this was the capability string:
    #   http://openconfig.net/yang/lldp/types?module=openconfig-lldp-types&revision=2016-05-16
    # then:
    #   openconfig-lldp-types
    # .. would be the module printed.
    # Scan capability string for module
    supported_model = re.search('module=(.*)&', capability)
    # If module found in string, store it.
    if supported_model is not None:
        # Module string was found, store it.
        print(f"Supported Model: {supported_model.group(1)}")
        # Store the module for later use.
        modules.append(supported_model.groups(0)[0])


# List of models that we want to download.
# We will get the schema for each and write it to disk.
models_desired = ['openconfig-extensions', 'openconfig-interfaces']
# Iterate each desired model and write it to ./models/
for model in models_desired:
    # Get the model schema.
    schema = m.get_schema(model)
    # Open new file handle.
    with open("./models/{}.yang".format(model), 'w') as f:
        # Write schema
        f.write(schema.data)

Augmentation

Pyang in most cases will combin the different models and augment the source.

# Example with dot1x without augmentation will not show much as the model depends on the other
pyang -f tree Cisco-IOS-XE-native.yang --tree-path=/native/dot1x

# Augmentation will add the added stuff to the original module (marking them with a prefix as given in the added (dot1x) module)
## E.g. prefix defined in dot1x.yang is "ios-dot1x" and container "guest-vlan" then becomes ios-dot1x:guest-vlan
pyang -f tree Cisco-IOS-XE-native.yang Cisco-IOS-XE-dot1x.yang --tree-path=/native/dot1x

Use Yang to generate Skeleton

Probably the most important usage of pyang here is the generation of template-xml, called skeletons. Either configure the device using regular CLI and extract the generated configuration using netconf-console or generate the stub/skeleton directly using pyang. For the skeleton generation, all needed modules must be available in the search path.

netconf-console --host 10.10.10.200 --port 830 --user admin --password xy --db running --get-config --xpath /native/dot1x

# Sample using Augmentatoin to generate sample xml skeleton for dot1x
pyang -f sample-xml-skeleton Cisco-IOS-XE-native.yang --sample-xml-skeleton-path=/native/dot1x

# Sample using augmentation to generate sample xml skeleton for logging
pyang -f sample-xml-skeleton Cisco-IOS-XE-native.yang Cisco-IOS-XE-logging.yang --sample-xml-skeleton-path=/native/logging

# See what is needed and the associated values (e.g. here uint16)
## Other useful option, --tree-depth=4, --tree-path=
pyang -f tree --tree-help
pyang -f tree Cisco-IOS-XE-native.yang Cisco-IOS-XE-logging.yang --tree-path=/native/logging
pyang -f tree Cisco-IOS-XE-native.yang Cisco-IOS-XE-logging.yang --tree-path=/native/logging/history

# Generate stub for data retrieval (default)
pyang -f sample-xml-skeleton Cisco-IOS-XE-native.yang Cisco-IOS-XE-logging.yang --sample-xml-skeleton-path=/native/logging/history

# Generate stub for Config object and adding type annotations
pyang -f sample-xml-skeleton Cisco-IOS-XE-native.yang Cisco-IOS-XE-logging.yang --sample-xml-skeleton-path=/native/logging/history --sample-xml-skeleton-annotations --sample-xml-skeleton-doctype=config

Note on Cisco Models

Some versions of the YANG models are not really stable, or better said don't pass yang validation/linting and therefore cannot be used to generate skeletons. E.g. with CSR and 17.3.4a we get the following errors if we try to generate an xml skeleton:

Cisco-IOS-XE-ip.yang:3546: error: "Cisco-IOS-XE-native:interface" in the path for Tunnel at Cisco-IOS-XE-native.yang:2269 (at Cisco-IOS-XE-ip.yang:3542) is not found
Cisco-IOS-XE-native.yang:2291 (at Cisco-IOS-XE-ipv6.yang:369): warning: node "Cisco-IOS-XE-native::native" is not found in module "Cisco-IOS-XE-ipv6"
Cisco-IOS-XE-native.yang:2588 (at Cisco-IOS-XE-logging.yang:1144): warning: node "Cisco-IOS-XE-native::native" is not found in module "Cisco-IOS-XE-logging"
sample-xml-skeleton plugin needs a valid module

When we do a lint of the corresponding module, we see that it contains many errors.

pyang --lint Cisco-IOS-XE-native.yang

In order to generate skeleton xmls, a stable version, like 17.1.1 should be used.
