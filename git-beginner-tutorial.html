Git

Intro

  • VCS (Version Control System)
    • Software that tracks and maanges changes to files over time
    • Compare / Undo / Share / Track changes / time-travel / Revert / combine changes / etc.
    • Git, Subversion, CVS, Mercurial
  • Git vs. Github
    • Git: VCS that runs locally (no account etc.), git repo local
    • Github: Account, internet, git repo in the cloud
  • Repo
    • = Workspace which tracks and manages files within a folder
    • Git tracks files in folders and subfolders
  • Commit
    • Snapshot / Checkpoint in time
    • Grouping of saved changes (changes to file1, file2 ... filen)
    • Intermediate step (add to staging area) before commiting to repo (.git folder updated)
    • Atomic: Keep each commit focused on a single thing (supports undo/rollback, easier to review)
    • Present tesne (imperative) recommended (not past tense)

Branching

  • General
    • Alternative timelines
    • Separate contexts (try new things, work in parallel)
  • HEAD & Refs
    • HEAD: Pointer that refers to current location.
      • HEAD is a reference to a branch pointer = Points to a particular branch
    • Branch = Reference to some commit xy
    • Analogy: Many bookmarks in a book, but only one of them can be active (head pointer) at a time
  • git branch, git switch, git checkout
  • Master & Main
    • default branch is master
    • Github switched from master to main in 2020
  • Switching with uncommited work
    • Requires stashing

  • General

Diff

  • General
    • Diffing Working Dir <> Staging Area
    • Diffing Branches
    • Diffing to common Ancestor Branch
    • Diffing Commits
    • Diffing specific files
    • git diff --staged
  • Default: List all changes in working dir that are NOT staged for next commit (Workdir <> Staging Area/Index)
  • Chunkg Header Format: @@ -start,count +start,count @@
    • Example: @@ -122,7 +123,7 @@ (start at line 122, remove 7 lines, start at line 123, add 7 lines)
    • Example: Also read as, @@ - part correspond to file A + part corresponds to file B@@
    • By default includes 3 lines of before & after to give some context
  • -- Operator used to separate paths from revisions/tree/branch
    • The -- separates the tree you want to check out from the files you want to check out.
    • Used for disambiguity, does not have to be used in all cases
    • Example: git diff master..feature -- master if there is a file called master in the repo (see links below)
    • Example: git diff master..feature somefile.txt works as no branch is called somefile.txt
  • Note: HEAD~1 means the parent of HEAD
command comment
git diff Staging Area (index) and Working Dir diff = what is not staged for next commit
git diff HEAD All changes in the working dir since last commit (staged and unstaged changes)
git diff HEAD path/to/a.txt Same as above, but only for a specific file
git diff --staged | git diff --cached All Changes between staging area and last commit = what will be included if commited now
git diff --staged a.txt Same as above, but only for a specific file
git diff -U0 NO context (default 3 lines), more readable
git diff master..feature Compare 2 branches (their commit) (not working dir)
git diff feat/1..feat/2 path/to/file Compare specific file of 2 branches
git diff feat/1..feat/2 -- path/to/file Same as above - Compare specific file of 2 branches
git diff master...feature Diff feature to common ancestor with master (if both have moved fwd)*
git diff master~2..HEAD Compare HEAD to master (not latest, but two revisions ago)
git diff master~2 Same as above
git diff commit_id_1..commit_id_2 Comparing changes using commit ids
git diff HEAD HEAD~1 New commit to the parent
git diff HEAD~1 HEAD Compare parent to the child

*three dots compares the top of the right branch (the HEAD) with the common ancestor of the two branches

Comments:

  • Line#1: Shows which files are being compared
  • Line#2: Metadata, Hashes of the files being compares followed by a mode
  • Line#3-4: Markers for file A (usually old version) and B (usually new version)
    • Can be differnt version if e.g. working on 2 branches with same parent (then A/B, not old/new or added/deleted)
  • Line#x-y: Chunks/Portions of changes, each chunk separated by @@, + context (3 lines before and after by default)
  • Line#x-y: @@-24,6 = From file A, 6 lines are extracted starting from line 24 (does NOT mean negative 24, - just refers to file A)
  • Line#x-y: +24,7@@ = From file B, 7 lines are extracted starting from line 24 (does NOT mean plus 24, + just refers to file B)
    • Note: Skip the line(s) marked with - for counting number of lines for second part
  • Ling#17: Line that begins with - comes from File A (red color does NOT equal deleted in all situations)
  • Line#18: Line that begins with + comes from File B (green color does NOT equal added in all situations)

Stashing

  • General
    • Usage Example: Swtich to Branch XY and have some uncommited/unfinished work, need to switch back to e.g. master for some reason.
    • In General, uncommited changes on one branch and switching to another branch. 2 Options for a result
      • Uncommited changes get moved to destination branch (assuming they do NOT conflict)
      • Git does NOT let one switch to destination branch if it detects a conflict (e.g. same lines modified)
    • Save, Pop, Apply
    • Similar like a Pause: Save changes without commiting them == stash uncommited changes avoids unnessary commits
      • Stashing helps in not destroying commit history.
    • Git stash will take ALL uncommited changes (staged & unstaged = Working Dir & Staging Area), save them and revert changes in working copy
  • Multiple Stashes Possible
  • WIP = Work in Progress
  • Other Options
    • Can also stash in one branch, and reapply to another branch.
command comment
git stash [save] Save ALL uncommited changes and stash them
git stash pop Reapply changes to working copy
git status Shows that working tree is clean after stashing
git stash list Show all different stashes
git stash show Shows stash at top = git stash show stash@{0}
git stash show stash@{1} Show a particular stash
git stash pop stash@{1} Reapply changes from different stash id to working copy
git stash apply Reapply changes wihtout removing from stash*
git stash drop Deletes last stash (=reference stash@{0})
git stash drop stash@{2} Deletes stash with particular stash ref
git stash clear Delete all stored stashes

*useful when applying to multiple branches

Undoing

  • General
    • Reset undos commit = moves the branch pointer (not the HEAD pointer) to a different commit (.e.g .git/refs/heads/develop)
      • = Modify git history and decide whether to keep the changes locally in working dir (soft/mixed) or discard the diff (hard)
      • per branch basis = if reset on feature = no impact on master, even if shared history
    • Restore restores file content (without moving HEAD) to a particular state (=content at a certain commit), default is restore to HEAD
      • Restore unstages changes from staging area/index (e.g. passwords.txt was added to staging area which should not ahve been done)
    • Checkout can do many things (create branch, switch to branch, restore files, undo history)
      • Checkout kind of overloaded, then git switch and git restore cmds where added. Whit these commands its clearer when HEAD pointer is moved (git switch) or not (git restore). With git checkout it sometimes is moved (checkout whole branch or commit-id) and sometimes not (checkout particular file)
    • revert creates a new commit which undos/reverses the changes from a commit
      • Can be helpful because can time-travel back and forth! With reset commit is lost forever.
      • Better when team colaboration because of commit history on other machines
  • Detached HEAD
    • Normally HEAD points to a branch reference (not a particular commmit). Branch reference then points to commit.
    • If new commit, branch reference gets updated to point to new commit. HEAD still points to branch reference.
    • Detached Head state results from time traveling back to a particular commit-id (git checkout ), does NOT happen e.g. if checking out a particular file at a certain commit ID, as only content of that file is changed.
    • Options if in detached HEAD state:
      • Stay in detached HEAD to view content of old commit, do experimantel changes
      • Go back to last branch (git switch -)
      • Create a new branch to make changes and save them (git switch -c <new_branch_name>)
        • Remember: Branch is based upon HEAD Pointer. So branch gets created at where HEAD points at that moment.
    • See state: cat .git/HEAD usually points to sth like refs/heads/master, if detached then commit-id
  • Restore, reset, revert, discarding changes, checking out particular commits
  • Reset options
    • --soft: Moves only the HEAD, Working dir remains intact, example: git reset --soft HEAD~
      • = Delete last/n commit(s) but keep changes in working dir & staging area
    • --mixed: Default, resets HEAD + moves everything to unstaged, Working dir remains intact, example: git reset [--mixed] HEAD~
      • = Delete last/n commit(s) but keep changes in working dir
      • Example: Made some commits, but on the wrong branch, but keep the work! git reset 1234567 && git switch -c feat/test && git add . && git commit -m 'on other branch'
    • --hard: resets HEAD + Index (staging area) + Working dir, example: git reset --hard HEAD~
      • = Delete last/n commit(s) + associated changes
  • Checkout a particular file or restoring it do NOT move the HEAD pointer!
command comment

General

  • git clone, git fetch, git pull, remote Tracking Branches
  • Remote = fancy name for destination URL + a label
  • Remote names
    • origin = convention, but does not need to be that at all
      • When repo cloned, default remote name given by clone command is called origin, but nothing special
      • No hidden meansings, abilities, similar to master/main for branches
  • Git push will create branch on remote if it does not exist
  • Upstream = "Link" that connects the local branch to a branch on e.g. GitHub (upstream)
    • Enables tracking of the upstream branch
    • Enables git push without having to specify
    • Git shall "remember" that local master branch is connected to master branch on origin
    • If not "connected" and doing git push, tells me branch xy has no upstream branch
  • Main vs Master
    • Renamed from master to main because term might offend (slavery), no technical background
  • Remotes and References
    • When cloning a Github/Gitlab (etc.) repo, an additional branch reference is created called Remote Tracking Branch
      • See ls .git/refs/remotes/origin/ in addition to
      • See ls .git/refs/heads/
    • Local Branch Pointer (can be moved via local commit etc) and
    • Remote Tracking Branch (cannot be moved locally directly)
      • Last time communication to remote repo happend, branch x on remote was pointint to that reference
      • Follows the syntax remote/branch, e.g. origin/master, upstream/master
      • Can be checked out/diffed like any other branch
  • Clonging a remote repo does not sync all branches, only main/master by default, even though content and pointers are synced
    • Git folder is synced and therefore git branch -r for remote knows all the branches, but not syned
    • Master/main automatically tracks remote/upstream branch, other branches do NOT
    • git switch <remote_branch_name/> locally, if this is detected by git switch, remote tracking is setup automatically
      • Condition: Same name on remote, e.g. origin/develop and then git switch develop will create local branch develop and set it up for tracking
  • Fetching and Pulling
    • git fetch: Downloads changes from remote repo but not integrated into workdir (see what others have doen, but don't mess with own work)
    • git pull: Downloads changes from remote repo and integrates into workdir (integrate others work) (updates HEAD)
      • git pull= git fetch (get latest changes from remote repo) + git merge (merge into workdir of current branch)
      • Where it is run from is important! Run on local branch develop but do git pull origin/main will incorporate main changes into develop
      • Can result in mege conflict
    • Fetch and pull will default to origin if not specified
    • Git pull branch will default to tracking branch of current branch

Collaboration Workflows

  • Pull Request (PR)
    • Not a feature of git, but GitHub, Bickbucket, Gitlab etc.
    • Review & discuss, approve, reject work to be merged into master/main
    • Request changes on review, approve
    • Together with branch protection rule powerful
  • Branch Protection Rules
    • Settings > Branches > Branch protection rules
    • Often used together with Pull requests to protect master/main

Centralized

  • Everybody works on single branch (master/main)
  • Good for single user, tiny team
  • Not a recommended workflow
  • Multiple collaborators: failed to push because tip of current branch is behind remote
    • Other collaborator pushed to remote repo since we last pulled
    • Pull changes down, merge (possible with conflict), then push up
  • Other problems
    • Collaborator XY cannot push up to share stuff if he is behind branch on remote
    • If code is incomplete/broken -> breaks the whole code base for everybode
    • Collaboration on feature with teammates only via incomplete code push to remote
    • Experiments/bug fixes always on "stable" code

Feature Branch

  • Nobody works on master/main
  • Master/main = project history with only stable code
  • Multiple teammates can collaborate on feature and share code without breaking master
  • Patterns
    • feat/xy, feat.xy, bug.xy, refactor.xy, etc.
  • Integrate via Pull Request or local merge into master/main and push back up
    • Pull requests prefereed for collaboration
  • Merge with conflict (either in online editor for simple merge or locally)
    • go into feature branch locally, merge in main, resolve conflict
      • git fetch origin, git switch feat/xy, git merge main and fix conflicts
    • go into main branch, merge in feature branch (we know no conflict can occur now that it was resolved on feature branch)
      • Merge with --no-ff to preserve history and create a new merge commit (not just move Head pointer to different commit)
      • git switch main, git merge --no-ff feat/xy, git push origin main

Forking

  • Common pattern for open source project where user is not a collaborator and does NOT have permission to push anything into project
